# 系统问题综合分析需求

## 背景
OpenClaw工作空间包含多个自动化金融投资任务（T01龙头战法、T99复盘扫描、T100宏观监控），近期出现多个重复问题影响系统稳定性和可靠性。需要MoA提供系统性解决方案。

## 问题列表（按优先级排序）

### 1. T100宏观监控原油数据获取失败（紧急）
**问题描述**：每日22:00宏观监控报告中原油数据始终显示"N/A"
**三层数据源同时失败**：
- **主源yfinance**：✅ 测试正常（BZ=F: $82.5, CL=F: $78.5）
- **备用源AKShare**：❌ `futures_zh_spot()` 抛出 `list index out of range` (AKShare库内部解析错误)
- **爬虫源playwright**：❌ 模块缺失（已安装但浏览器安装中）
- **fallback方案**：调用openclaw `web_fetch` 命令失败

**影响**：宏观报告缺少重要大宗商品价格数据
**已实施修复**：
- 创建Cron环境诊断工具，发现playwright缺失、AKShare接口bug
- 添加AKShare异常处理（捕获IndexError/KeyError/ValueError）
- 安装playwright包（浏览器安装中）

### 2. T99复盘扫描引擎JSON序列化错误（高优先级）
**问题描述**：T99每日14:30扫描连续多日失败，错误：`TypeError: Object of type bool is not JSON serializable`
**根本原因**：`short_term_signal_engine`返回的数据包含`numpy.bool_`类型
**影响**：即使扫描逻辑成功，也无法输出JSON结果，导致整体失败
**位置**：`/root/.openclaw/workspace/skills/a-share-short-decision/main.py` 第18行 `_print`函数
**已实施修复**：已添加`EnhancedJSONEncoder`处理numpy类型，但可能需要更完善

### 3. T99扫描引擎API超时问题（高优先级）
**问题描述**：`get_sector_rotation()`函数在非交易时段API挂起
**错误模式**：扫描卡在"Running A-share short-term signal engine for YYYY-MM-DD (timeout 900s)..."
**根本原因**：
- Tushare `moneyflow_ind_dc` API在非交易时段无限等待
- `trade_cal`交易日检查自身可能挂起
- 连续多个交易日失败（3月2日-3月6日）

**已实施修复**：
- 添加周末检查（周六/日直接返回空数据）
- 创建自动诊断工具`t99_diagnostic_tool.py`
- 但超时问题仍持续

### 4. 磁盘空间管理问题（中优先级）
**问题描述**：系统磁盘使用率80% (32G/40G)
**风险**：可能影响新软件安装、日志写入
**位置**：主要占用/tmp/日志、任务历史数据
**建议**：需要定期清理策略

### 5. 系统内部执行失败噪音（低优先级）
**问题描述**：频繁出现`Exec failed (session-name-fragment, code 0)`消息
**特征**：短会话名称片段（如`plaid-lo`, `delta-pr`等），退出码0
**理解**：OpenClaw内部会话管理机制产生的日志噪音，非错误
**影响**：低，但产生大量日志条目，可能掩盖真正错误

## 约束条件
1. **免费/低成本优先**：解决方案应优先使用免费或低成本组件
2. **系统健壮性**：重要数据源应有至少两层备用方案
3. **自动化修复**：避免手动干预，实现自我修复能力
4. **维护成本**：解决方案应易于维护，不增加过多技术债务
5. **环境一致性**：确保cron环境与交互式环境一致

## MoA分析要求
针对上述每个问题，请提供：

- 按紧急程度、影响范围、实施成本排序
- 建议的实施时间表
- 可能的依赖关系和实施顺序

- 如何预防类似问题再次发生
- 建立哪些健康检查和预警机制
- 如何实现问题自动检测和修复
